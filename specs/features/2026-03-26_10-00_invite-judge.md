# Feature: Invite Judge

## Problem Statement

Admins need a frictionless way to get judges into an event. Judges are typically bootcamp mentors, industry guests, or company reps — they won't create accounts ahead of time. The invite flow must be zero-friction: share a link (or show a QR code), the judge fills in their name/role/company, and they're in.

---

## User Stories

- **As an admin**, I want to generate a shareable invite link (and QR code) for my event so that I can onboard judges without managing individual accounts.
- **As a judge**, I want to click an invite link, enter my details, and immediately access the event — without signing up or remembering a password.
- **As an admin**, I want to prevent duplicate judges (by name) from joining the same event so that the judge list stays clean.
- **As a judge**, if I already joined this event, I want to be told so — not silently given a second entry.

---

## Acceptance Criteria

### Backend

- [ ] `password` column on `users` table becomes nullable (migration)
- [ ] `users.role` enum extended to include `'judge'` (migration)
- [ ] `events` table gains a `judge_judge_invite_token` column (UUID, unique, nullable) (migration)
- [ ] New `event_judges` join table: `id`, `event_id` (FK), `user_id` (FK), `created_at`, `updated_at` — unique constraint on `(event_id, user_id)` (migration)
- [ ] `POST /api/events/:eventId/judge-invite` — generates (or regenerates) an `judge_invite_token` for the event; returns the token. Requires auth + admin role.
- [ ] `GET /api/events/:eventId/judge-invite` — returns the current invite token (or null). Requires auth + admin role.
- [ ] `GET /api/judge-invite/:token` — public route (no auth). Validates the token, returns event name and id (so the judge registration page can show context).
- [ ] `POST /api/judge-invite/:token/join` — public route (no auth). Accepts `{ name, role, company }`. Creates a User with `role: 'judge'` (no password), creates an EventJudge link, issues a JWT. Returns `{ token (JWT), user, event }`.
- [ ] Duplicate prevention: if a judge with the same `name` (case-insensitive) already exists for the event, return 409 with a clear error message.
- [ ] JWT payload for judges includes `{ id, name, role: 'judge' }` (no email).
- [ ] Input validation on join: `name` required (2-50 chars), `role` required (2-50 chars), `company` required (2-100 chars).

### Frontend

- [ ] **Invite Section** on EventDetailPage: a collapsible or always-visible section where the admin can generate/view the invite link and QR code.
- [ ] "Generate Invite Link" button — calls `POST /api/events/:eventId/judge-invite`, then displays the link + QR code.
- [ ] If an invite token already exists, show the link + QR code immediately on page load.
- [ ] "Copy Link" button next to the invite URL — copies to clipboard with brief confirmation feedback.
- [ ] QR code rendered client-side from the invite URL (use a lightweight library like `qrcode.react`).
- [ ] **Judge Join Page** (`/judge-invite/:token`) — public page (no auth required):
  - Shows event name as heading for context.
  - Form with three fields: Name, Role (e.g. "CTO", "Senior Developer"), Company.
  - Inline validation on all fields.
  - On submit: calls `POST /api/judge-invite/:token/join`.
  - On success: stores JWT + user in AuthContext, navigates to `/events/:eventId`.
  - On 409 (duplicate name): shows inline error — "A judge with this name has already joined this event."
  - On invalid/expired token: shows a clear error state — "This invite link is not valid."
- [ ] **Event page as Judge role**: when a judge navigates to `/events/:eventId`, they see the event details and team list in read-only mode (no edit/delete buttons, no criteria management, no invite section). Scoring UI is out of scope for this feature.

## UI/UX Notes

### Invite Section (Admin — EventDetailPage)

- Placed after the existing sections (Teams, Criteria).
- States:
  - **No invite yet**: Shows a "Generate Invite Link" button.
  - **Invite active**: Shows the full invite URL in a read-only input, a "Copy Link" button, and the QR code rendered below.
- QR code size: ~200x200px, with enough padding for easy scanning.
- The invite URL format: `{CLIENT_URL}/judge-invite/{token}`

### Judge Join Page

- Clean, centered layout — similar visual weight to the Login page.
- Shows the event name prominently so the judge knows they're in the right place.
- States:
  - **Default**: Form with Name, Role, Company fields + "Join as Judge" button.
  - **Loading**: Button shows loading state, form disabled.
  - **Error (validation)**: Inline field errors.
  - **Error (duplicate)**: Alert/banner above form.
  - **Error (invalid token)**: Full-page error state replacing the form — "This invite link is not valid or has expired."
  - **Success**: Redirect to event page (no success screen needed).

### Event Page (Judge View)

- Same layout as admin EventDetailPage but **read-only**:
  - Team cards displayed without edit/delete actions.
  - No "Add Team" button.
  - No Criteria section (judges don't need to see criteria definitions — they'll see them during scoring in a future feature).
  - No Invite section.
- Header should show the judge's name (from AuthContext) instead of the admin email.

### Responsive Behavior

- QR code scales down on mobile but remains scannable (min ~150px).
- Judge Join form is single-column on all breakpoints.
- Copy Link button works on mobile (Clipboard API with fallback).

## Data Model Changes

### Migration: Make password nullable

```
ALTER TABLE users ALTER COLUMN password DROP NOT NULL;
```

### Migration: Add 'judge' to users.role enum

```
ALTER TYPE enum_users_role ADD VALUE 'judge';
```

### Migration: Add judge_invite_token to events

```
ALTER TABLE events ADD COLUMN judge_invite_token UUID UNIQUE;
```

### Migration: Create event_judges table

```sql
CREATE TABLE event_judges (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  event_id UUID NOT NULL REFERENCES events(id) ON DELETE CASCADE,
  user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  UNIQUE(event_id, user_id)
);
```

### Model: User (updated)

- `password`: STRING, **allowNull: true**
- `role`: ENUM('admin', 'judge')
- Add: `name` (STRING, allowNull: true — only judges have it)
- Add: `company` (STRING, allowNull: true — only judges have it)
- Add: `title` (STRING, allowNull: true — this is the judge's "role/title", named `title` to avoid collision with the `role` column)

### Model: EventJudge (new)

- `id`: UUID, PK
- `event_id`: UUID, FK -> events
- `user_id`: UUID, FK -> users
- Unique constraint on (event_id, user_id)

### Model: Event (updated)

- Add: `judge_invite_token`: UUID, unique, allowNull: true

## API Endpoints

| Method | Path | Auth | Description |
|--------|------|------|-------------|
| POST | `/api/events/:eventId/invite` | Admin | Generate/regenerate invite token |
| GET | `/api/events/:eventId/invite` | Admin | Get current invite token |
| GET | `/api/invite/:token` | Public | Validate token, return event info |
| POST | `/api/invite/:token/join` | Public | Judge submits details, gets JWT |

### POST `/api/events/:eventId/invite` — Response

```json
{
  "data": { "judge_invite_token": "a1b2c3d4-..." },
  "error": null
}
```

### GET `/api/invite/:token` — Response

```json
{
  "data": { "event": { "id": "...", "name": "Demo Day 2026" } },
  "error": null
}
```

### POST `/api/invite/:token/join` — Request

```json
{ "name": "Jane Smith", "title": "CTO", "company": "Acme Corp" }
```

### POST `/api/invite/:token/join` — Response (201)

```json
{
  "data": {
    "token": "jwt...",
    "user": { "id": "...", "name": "Jane Smith", "role": "judge" },
    "event": { "id": "...", "name": "Demo Day 2026" }
  },
  "error": null
}
```

### POST `/api/invite/:token/join` — Response (409)

```json
{
  "data": null,
  "error": "A judge with this name has already joined this event"
}
```

## Technical Notes

- **QR code library**: `qrcode.react` (lightweight, React-native, no server dependency).
- **Invite token**: Generated server-side with `crypto.randomUUID()`. Stored on the Event row. One token per event — regenerating replaces the old one.
- **Auth flow for judges**: The `POST /api/judge-invite/:token/join` endpoint creates the User + EventJudge records and returns a JWT. The client stores this in AuthContext exactly like admin login. The existing `auth` middleware and `requireRole()` work as-is for judges.
- **Judge's "role" field vs User.role column**: The judge enters their professional title (e.g., "CTO"). This is stored as `title` on the User model to avoid naming collision with the `role` column (which holds `'admin'` or `'judge'`).
- **Duplicate check**: On join, query for an existing User with the same name (case-insensitive, trimmed) who is linked to this event via EventJudge. If found, return 409.
