## Feature: Team Scoring

### Problem Statement

Judges can view teams and criteria on the event page, but have no way to actually score them. The core purpose of the platform — collecting structured evaluations from judges — is not yet functional. Without scoring, the judging workflow is incomplete.

### User Stories

- As a **judge**, I want to score a team on each criterion (1–5) and leave an optional comment, so that my evaluation is recorded for the event.
- As a **judge**, I want to see my previously submitted scores when I revisit a team, so that I know what I already scored and can update if needed.
- As a **judge**, I want to update my scores for a team after submission, so that I can change my mind before results are finalized.

### Data Model

#### New Model: `Score`

| Column        | Type         | Constraints                                      |
|---------------|--------------|--------------------------------------------------|
| `id`          | UUID         | PK, auto-generated                               |
| `event_id`    | UUID         | FK → events, NOT NULL, CASCADE delete             |
| `team_id`     | UUID         | FK → teams, NOT NULL, CASCADE delete              |
| `criterion_id`| UUID         | FK → criteria, NOT NULL, CASCADE delete           |
| `judge_id`    | UUID         | FK → users, NOT NULL, CASCADE delete              |
| `value`       | INTEGER      | NOT NULL, CHECK (value >= 1 AND value <= 5)       |
| `created_at`  | TIMESTAMP    |                                                   |
| `updated_at`  | TIMESTAMP    |                                                   |

**Unique constraint:** `(team_id, criterion_id, judge_id)` — one score per judge per criterion per team.

#### New Model: `TeamComment`

| Column      | Type    | Constraints                                      |
|-------------|---------|--------------------------------------------------|
| `id`        | UUID    | PK, auto-generated                               |
| `event_id`  | UUID    | FK → events, NOT NULL, CASCADE delete             |
| `team_id`   | UUID    | FK → teams, NOT NULL, CASCADE delete              |
| `judge_id`  | UUID    | FK → users, NOT NULL, CASCADE delete              |
| `comment`   | TEXT    | NOT NULL                                          |
| `created_at`| TIMESTAMP |                                                 |
| `updated_at`| TIMESTAMP |                                                 |

**Unique constraint:** `(team_id, judge_id)` — one comment per judge per team.

**Why separate from Score?** The comment is per-team (not per-criterion), and it's optional — keeping it in its own table avoids nullable columns and keeps the Score model clean.

### API Endpoints

#### Submit/Update Scores for a Team

```
PUT /api/events/:eventId/teams/:teamId/scores
```

**Auth:** Judge (must be assigned to this event)

**Request body:**
```json
{
  "scores": [
    { "criterionId": "uuid", "value": 3 },
    { "criterionId": "uuid", "value": 5 }
  ],
  "comment": "Great demo, loved the UI"   // optional — omit or null to skip
}
```

**Validation:**
- `scores` array must include an entry for every criterion in the event (mandatory, all-at-once)
- Each `value` must be an integer between 1 and 5
- `criterionId` values must all belong to the given event
- Judge must be associated with the event via `event_judges`

**Behavior:**
- Upsert: creates scores if none exist, updates if they do (within a transaction)
- If `comment` is provided and non-empty, upsert the TeamComment record
- If `comment` is null/omitted and a TeamComment exists, delete it (judge removed their comment)

**Response:**
```json
{
  "data": {
    "scores": [
      { "criterionId": "uuid", "value": 3 },
      { "criterionId": "uuid", "value": 5 }
    ],
    "comment": "Great demo, loved the UI"
  },
  "error": null
}
```

#### Get Judge's Scores for a Team

```
GET /api/events/:eventId/teams/:teamId/scores
```

**Auth:** Judge (must be assigned to this event)

**Response (scores exist):**
```json
{
  "data": {
    "scores": [
      { "criterionId": "uuid", "value": 3 },
      { "criterionId": "uuid", "value": 5 }
    ],
    "comment": "Great demo, loved the UI"
  },
  "error": null
}
```

**Response (no scores yet):**
```json
{
  "data": {
    "scores": [],
    "comment": null
  },
  "error": null
}
```

### UI/UX Flow

#### Entry Point: Team Card on Event Detail Page (Judge View)

- Each **TeamCard** gets a "Score Team" button (or "Edit Scores" if the judge has already scored).
- Clicking the button opens a **scoring modal/panel** for that team.
- A visual indicator on the TeamCard shows whether the judge has already scored this team (e.g., a checkmark badge or "Scored" label).

#### Scoring Modal

**Header:**
- Team name
- Close button (X)

**Body — Criteria List:**
For each criterion (displayed in `sort_order` order):
- Criterion **name** (bold)
- Criterion **description** (secondary text below name)
- **Score selector**: 5 clickable options (1–5), styled as a button group or star-like selector
  - Unselected state: outlined / muted
  - Selected state: filled / highlighted
  - Labels visible: "1" through "5" (optionally with "Lowest" / "Highest" at the ends)

**Comment Section (below criteria):**
- Label: "Comment (optional)"
- Textarea, 3–4 rows, with placeholder: "Share your thoughts on this team's demo..."
- Character limit: 1000 characters (with counter shown when > 800 chars typed)

**Footer:**
- "Submit Scores" button (primary) — disabled until all criteria have a score selected
- "Cancel" button (secondary) — closes modal, discards unsaved changes

#### States

| State | Behavior |
|-------|----------|
| **Default (unscored)** | All score selectors empty, submit button disabled |
| **Partially filled** | Some criteria scored — submit button remains disabled, unfilled criteria subtly highlighted |
| **All criteria scored** | Submit button enabled |
| **Submitting** | Submit button shows loading state, form inputs disabled |
| **Success** | Modal closes, TeamCard updates to show "Scored" indicator, brief success toast |
| **Error** | Inline error message above the submit button, form remains open, inputs stay populated |
| **Edit mode** | Modal opens pre-filled with existing scores and comment, submit button says "Update Scores" |

#### Responsive Behavior

- **Desktop:** Centered modal overlay (max-width ~500px)
- **Mobile:** Full-screen sheet sliding up from bottom, or full-width modal

### Acceptance Criteria

- [ ] New `scores` table exists with correct schema, constraints, and indexes
- [ ] New `team_comments` table exists with correct schema, constraints, and indexes
- [ ] `PUT /api/events/:eventId/teams/:teamId/scores` validates all criteria are present and values are 1–5
- [ ] Scores are upserted (create or update) within a single transaction
- [ ] Comments are upserted when provided, deleted when cleared
- [ ] `GET /api/events/:eventId/teams/:teamId/scores` returns the judge's own scores only
- [ ] Both endpoints reject requests from judges not assigned to the event
- [ ] TeamCard shows "Score Team" button for judges, with "Scored" indicator when scores exist
- [ ] Scoring modal displays all criteria in sort_order with name and description
- [ ] Score selector enforces 1–5 range, all criteria must be scored before submission
- [ ] Comment textarea is optional with 1000-char limit
- [ ] Submit button is disabled until all criteria are scored
- [ ] Previously submitted scores pre-populate the modal on reopen
- [ ] Success: modal closes, card updates, toast shown
- [ ] Error: inline message, form stays open with data intact
- [ ] Mobile-responsive modal layout

### Out of Scope

- Results/leaderboard/aggregation screen
- Event status gating (scoring availability based on draft/active/completed)
- Score locking or finalization
- Admin viewing individual judge scores
- Weighted criteria (all criteria have equal weight)
- Bulk scoring (scoring multiple teams at once)
